TLS
Transport Layer Security
TLS creates an encrypted channel over the network, ensuring data cannot be eavesdropped or tampered with in transit. It is the successor to SSL and is used in HTTPS, MQTT over TLS, OPC UA, and more.
Core Protocol
🤝
mTLS
Mutual TLS
Standard TLS only verifies the server. mTLS also requires the client to present a certificate, confirming the connecting device is authorized. Strongly recommended for IoT production deployments.
Advanced Security
🏛️
Root CA
Certificate Authority
A trusted third party that signs certificates with its private key. Anyone trusting the same Root CA automatically trusts all certificates it issued — no manual setup needed per device.
PKI Infrastructure
✍️
Self-Signed Certificate
No CA Involvement
Signed by the entity's own private key, without any CA. Fast to deploy, but both parties must manually exchange and import each other's certificate. Becomes unmanageable at scale.
Simple Deployment
📄
CSR
Certificate Signing Request
The applicant packages their public key and identity info into a CSR and submits it to the CA. The CA signs it to produce an official certificate. The private key never leaves the applicant.
Certificate Request
🔑
Session Key
Temporary Symmetric Key
After the handshake, both sides derive the same temporary symmetric key. All data is encrypted with it — much faster than asymmetric encryption. Each new connection creates a unique Session Key.
Symmetric Encryption
👋
TLS Handshake
Connection Negotiation Flow
① Client Hello → ② Server presents certificate → ③ Client verifies → ④ (mTLS) Client presents certificate → ⑤ Both derive Session Key → ⑥ Encrypted communication begins.
Core Flow
⚖️
One-way vs. Mutual Auth
Authentication Scope
One-way: only the device verifies the server — "am I connecting to the right server?" Mutual (mTLS): server also verifies the device — "is this an authorized device?" Use mutual in production.
